## 1. Project Overview

**NutriCan** is a mobile-first progressive web app (PWA) designed to provide personalised nutritional guidance specifically tailored for **cervical cancer patients** in the Ugandan/East African context. The app combines AI-driven meal planning, food safety analysis, health journaling, symptom support, and simulated telehealth consultations — all presented within a polished, offline-capable interface.

### Core Features

| Feature | Description |
|---|---|
| **AI Meal Plans** | 7-day personalised Ugandan meal plans generated by the NutriCan ML Model |
| **Food Safety Checker** | Instant AI-powered food safety analysis based on the user's diagnoses |
| **Nutrient Tracker** | Log meals with calorie, sugar, and salt data retrieved from AI |
| **Health Journal** | Track daily weight, energy levels (1–10), and blood pressure readings |
| **Symptom Tips** | Curated food recommendations for common cancer treatment side effects |
| **Live AI Doctor** | Simulated consultation chat with AI doctor personas |
| **Document Upload** | Medical document submission stored securely on DigitalOcean Spaces |
| **Guest Portal** | Limited read-only browsing without account creation |
| **Freemium Plans** | Free, 7-day trial, and Premium subscription tiers |

**Communication pattern:**  
All frontend → backend communication is via JSON REST. The frontend stores a JWT in `localStorage` and sends it as `Authorization: Bearer <token>` on every authenticated call. The backend verifies every token server-side before processing any request.

### 3.5 `App.tsx` — Root Orchestrator

`App.tsx` is the single top-level component. It owns:

- **Session state**: `currentPage`, `userProfile`, `authInitialView`
- **Session restoration**: On mount, calls `db.getSession()` which validates the stored JWT against `/api/auth/me`. If valid, jumps straight to `dashboard`; otherwise goes to `splash`.
- **Lazy loading**: All six page components are loaded via `React.lazy()` wrapped in `<Suspense>`. This splits each screen into its own JavaScript chunk, reducing initial load size.
- **Routing**: A `switch` statement on `currentPage` renders the correct component.
- **Guest handling**: Guest users (identified by `profile.isGuest === true`) are routed to `<GuestPortal>` instead of `<Dashboard>`, completely isolating their experience.

#### Page Transition Flow

```
null (loading) → splash → terms → onboarding → auth → dashboard
                                                     → guest (isGuest)
                          auth ←────────────── logout
```

#### Key Callbacks

| Callback | What it does |
|---|---|
| `handleAuthSuccess(profile)` | Receives profile after login; routes to `guest` or `dashboard` |
| `handleLogout()` | Calls `db.signOut()`, clears state, navigates to `auth` (sign-in view) |
| `handleGuestSignUp()` | Called from GuestPortal's "Sign Up" CTA; clears guest state, goes to `auth` |

### 3.7 Components

#### `SplashScreen.tsx`
Animated landing screen with the NutriCan logo and brand colours. Accepts `onGetStarted` callback.

#### `TermsScreen.tsx`
Full-screen scrollable terms and conditions display. Requires explicit "I Agree" before proceeding. Accepts `onAgree` and `onBack`.

#### `OnboardingScreen.tsx`
Multi-step carousel that introduces the app's key features with illustrations. Accepts `onComplete` and `onBack`.

#### `AuthScreen.tsx`

The most complex standalone component. Manages three internal views:

- **`initial`**: Landing card with buttons for "Create Account", "Sign In", and "Continue as Guest".
- **`signIn`**: Single-step form (email + password) that calls `db.signIn()`.
- **`signUp`**: **Two-step form:**
  - **Step 1** — Basic info: name, age, height, weight, email, password (min 6 chars).
  - **Step 2** — Medical profile: cancer stage, co-morbidities with sub-type dropdowns (e.g., Diabetes Type 1/2), treatment stages.

Co-morbidities are stored as formatted strings: `"Diabetes (Type 2)"`, `"Hypertension (Stage 1)"`, enabling readable display without separate lookup tables.

Guest login calls `db.signInAnonymously()` first (creates a throwaway backend record), falling back to a local guest profile if the backend is unreachable.

#### `Dashboard.tsx`

The largest file in the project. Renders all post-authentication functionality. Internal architecture:

**Reusable sub-components (defined inline):**
- `Modal` — Glassmorphic overlay with close button and scroll containment.
- `BottomNavBar` — Fixed bottom navigation with premium lock logic.

**Home Page features:**
- Weekly meal plan cards (AI-generated, with swap functionality)
- Food safety checker input
- Quick symptom tips section

**Tracker Page features:**
- `ComposedChart` (Recharts) showing weight and energy over 30 journal entries
- Nutrient log with calorie/sugar/salt totals
- Log meal form with AI nutrient retrieval

**Live Page:**
- Real-time AI doctor consultation using chat interface
- AI persona selection (doctor profiles)
- System health report builder that sends context to the AI

**Library Page** (Premium):
- Educational resources on nutrition, treatment side effects, and recovery

**Profile Page:**
- Editable user stats (weight, height)
- Plan management and trial countdown
- Document upload trigger
- Logout button

**Audio utilities (inside Dashboard.tsx):**
`decodeBase64`, `encodeBase64`, `decodeAudioData` — used for any future voice/audio features within the Live session. The `decodeAudioData` function converts raw PCM Int16 bytes into a browser-native `AudioBuffer` using the Web Audio API.

#### `GuestPortal.tsx`

A completely self-contained portal. It:
1. Generates an AI meal plan on mount using a default guest `UserProfile`.
2. Renders a read-only meal plan view.
3. Intercepts all interactive actions (meal swap, food checker, etc.) and shows a `SignUpModal`.

The `SignUpModal` lists four premium benefits and offers a "Create Free Account" CTA that calls `onSignUp`, which routes back to the main auth flow in `App.tsx`.

### 3.8 Services Layer

#### `services/config.ts`

Single export: `API_BASE_URL`.

```typescript
export const API_BASE_URL =
  (import.meta as any).env?.VITE_API_URL ?? 'http://localhost:4000';
```

In development: resolves to `http://localhost:4000`.  
In production: reads from the `VITE_API_URL` environment variable set in the hosting environment (e.g., Vercel, Netlify).

#### `services/db.ts` — API Client

Replaces a former Firebase/Firestore layer with a clean JWT-based REST client. All components interact with the backend exclusively through this module.

**Token management:**
```typescript
const TOKEN_KEY = 'nutrican_token';
// stored in localStorage
saveToken(token)   // on login
getToken()         // on every authenticated request
clearToken()       // on logout
```

**`apiFetch<T>(path, options)`** — Low-level fetch wrapper:
- Automatically injects `Authorization: Bearer <token>` when `auth: true` is passed.
- Calls `response.json()` and throws an `Error` if `response.ok` is false, extracting the server's `message` field.
- Returns typed `T` for type-safe call sites.

**`mapProfile(raw)`** — Maps raw API response to the `UserProfile` TypeScript interface, providing safe defaults for optional fields.

**Public `db` object methods:**

| Method | HTTP Call | Description |
|---|---|---|
| `db.signUp(email, pass, profile)` | `POST /api/auth/signup` | Creates account, saves JWT |
| `db.signIn(email, pass)` | `POST /api/auth/signin` | Logs in, saves JWT |
| `db.signInAnonymously()` | `POST /api/auth/guest` | Creates throwaway guest account |
| `db.signOut()` | (local) | Clears JWT from localStorage |
| `db.getSession()` | `GET /api/auth/me` | Validates stored JWT, returns profile |
| `db.saveProfile(profile)` | `PATCH /api/profile` | Updates all profile fields |
| `db.updateProfile(partial)` | `PATCH /api/profile` | Partial profile update |
| `db.getJournalEntries()` | `GET /api/journal` | Fetches last 30 entries |
| `db.addJournalEntry(entry)` | `POST /api/journal` | Creates journal entry |
| `db.getLoggedMeals()` | `GET /api/meals` | Fetches last 50 logged meals |
| `db.logMeal(name, nutrients)` | `POST /api/meals` | Logs a meal with nutrient data |
| `db.uploadMedicalDocs(file)` | `POST /api/documents/upload` | Uploads PDF to DigitalOcean Spaces |
| `db.getDocuments()` | `GET /api/documents` | Lists documents with signed download URLs |
| `db.upgradeToPremium()` | `POST /api/profile/upgrade` | Upgrades plan to 'Premium' |

#### `services/geminiService.ts` — AI Client

Despite the filename (legacy from a prior Gemini implementation), all AI calls now go to the Express backend's `/api/ai/*` routes, which in turn call the **NutriCan ML Model**.

**`aiPost<T>(endpoint, body)`** — Shared helper:
- Reads `nutrican_token` from localStorage.
- POSTs to `${API_BASE_URL}/api/ai/${endpoint}` with JSON body and Bearer token.

**Food Image Mapping:**  
`FOOD_IMAGE_MAP` is a large static dictionary mapping ~70 Ugandan food keywords to Unsplash/Wikipedia image URLs. The `getMealPhotoUrl(mealName)` function:
1. Lowercases the meal name.
2. Sorts keys by length (longest first) to ensure compound names like `"sweet potato"` match before `"potato"`.
3. Returns the matching image URL or a healthy-plate fallback.

**Exported AI functions:**

| Function | AI Endpoint | Input | Output |
|---|---|---|---|
| `checkFoodSafety` | `/api/ai/food-safety` | `foodName`, `userProfile` | `FoodSafetyResult { status, reason }` |
| `generateMealPlan` | `/api/ai/meal-plan` | `userProfile` | `WeeklyMealPlan` (7 days × 3 meals) |
| `swapMeal` | `/api/ai/swap-meal` | `userProfile`, `mealToSwap`, `day`, `mealType` | Single `Meal` object |
| `getNutrientInfo` | `/api/ai/nutrient-info` | `mealName` | `NutrientInfo { calories, sugar, salt }` |
| `getSymptomTips` | `/api/ai/symptom-tips` | `symptom` (SymptomType) | `RecommendedFood[]` |
| `getDoctorChatResponse` | `/api/ai/doctor-chat` | `doctor`, `userProfile`, `history`, `newMessage` | `string` reply |

All functions have a `try/catch` that returns a safe fallback value on error so the UI degrades gracefully without crashing.

### 3.9 Context API — `ThemeContext`

`ThemeContext` provides global dark/light mode across the entire app.

**State**: `theme: 'light' | 'dark'` — initialised from `localStorage.getItem('theme')`, defaults to `'dark'`.

**Effect**: On every `theme` change, adds `'dark'` or `'light'` as a class to `document.documentElement`, which activates Tailwind's `dark:` variant. The preference is persisted to `localStorage`.

**`ThemeProvider`** wraps `<App />` at the root. All components access the theme via `useContext(ThemeContext)`.

### 4.3 Server Entry Point (`server.js`)

The server starts in two phases to ensure health checks always pass:

1. **HTTP server starts first** on `PORT` (default `4000`), immediately accepting requests.
2. **MongoDB connection is attempted asynchronously** after the server is already listening. If it fails, the error is logged and the connection module retries automatically — the health check endpoint remains reachable throughout.

```
app.listen(PORT, () => {
  connectDB(process.env.MONGODB_URI).catch(...);  // non-blocking
});
```

**CORS configuration:**
- Development: Any `http://localhost:{port}` origin is allowed automatically.
- Production: Only origins listed in `ALLOWED_ORIGINS` (comma-separated env var) are allowed.
- Requests with no `Origin` header (mobile apps, Postman, curl) are always allowed.

**Global error handler:**
A final `app.use((err, req, res, next) => ...)` middleware catches any uncaught errors and responds with `{ message }` and the appropriate HTTP status code.

**Graceful shutdown:**
`SIGTERM` and `SIGINT` signal handlers call `server.close()` then `mongoose.disconnect()` before `process.exit(0)`. This ensures in-flight requests complete and the database connection is cleanly released.

**Health check route:**
```
GET /health → { status: 'ok', db: 'connected' | 'disconnected' }
```
Used by DigitalOcean App Platform and other PaaS health monitors.

### 4.4 Database Layer

#### `db/connection.js`

**DNS configuration (Windows fix):**
```javascript
dns.setServers(['8.8.8.8', '8.8.4.4', '1.1.1.1']);
dns.setDefaultResultOrder('ipv4first');
```
Forces Node.js's c-ares DNS resolver to use public DNS servers and prefer IPv4, resolving a common `ENOTFOUND` issue on Windows development machines.

**Connection options:**
```javascript
mongoose.connect(uri, {
  serverSelectionTimeoutMS: 10_000,  // Initial connection timeout
  socketTimeoutMS: 45_000,           // Idle socket timeout
  maxPoolSize: 10,                   // Connection pool size
});
```

**Index creation:**
On first successful connect, `ensureIndexes()` runs `Model.createIndexes()` on all four models. This is idempotent and ensures compound indexes exist without needing to manage migrations.

Development-only script. **Aborts if `NODE_ENV === 'production'`.**

Clears `User`, `JournalEntry`, and `Meal` collections, then creates:
- One demo user: `demo@nutrican.app` / `password123`
- 7 sample journal entries
- 5 sample logged meals

Run with: `npm --prefix backend run seed`

```javascript
  cancerStage:        String (required),

  treatmentStages:    [String] (default: []),
  plan:               enum['Free', 'Premium'] (default: 'Free'),
  isVerified:         Boolean (default: false),
  documentsSubmitted: Boolean (default: false),
  isGuest:            Boolean (default: false),
  trialStartedAt:     Date (optional),
  bmi:                Number (optional),
}
```

**Pre-save hook:** Before every save, if `passwordHash` has been modified (and is not empty), it is run through `bcrypt.hash(password, 12)`. This means the route handlers store the plain password in `passwordHash` and the hook transparently replaces it with the hash.

**Instance method:** `user.comparePassword(plain)` → calls `bcrypt.compare(plain, this.passwordHash)`.

**JSON serialisation:** `toJSON` transform deletes `passwordHash` from all API responses, so the hash is never exposed.

**Timestamps:** Mongoose `{ timestamps: true }` adds `createdAt` and `updatedAt` automatically.

### 4.6 Authentication Middleware

Located at `middleware/auth.js`.

#### `requireAuth` Express Middleware

Applied to all routes that need authentication. Execution flow:

1. Reads `Authorization` header, expects format `Bearer <token>`.
2. If missing or malformed → `401 Authorization token missing.`
3. Calls `jwt.verify(token, process.env.JWT_SECRET)` to decode and validate the token.
4. Looks up the user in MongoDB by `decoded.sub` (the user's `_id`).
5. If user no longer exists → `401 User no longer exists.`
6. If JWT is expired or tampered → `401 Invalid or expired token.`
7. On success: attaches the full `User` document to `req.user` and calls `next()`.

This pattern means every authenticated handler automatically has fresh user data from the database.

#### `signToken(user)` Helper

```javascript
jwt.sign({ sub: user._id }, process.env.JWT_SECRET, {
  expiresIn: process.env.JWT_EXPIRES_IN || '7d',
});
```

Signs a compact JWT with the user's `_id` as the `sub` claim. Default expiry is 7 days (configurable via `JWT_EXPIRES_IN`).

### 4.7 API Routes

#### `POST /api/auth/signup`

Validates email uniqueness, minimum password length (6 chars), then creates a `User` document. The pre-save hook hashes the password automatically. Returns `{ token, profile }`.

#### `POST /api/auth/signin`

Finds user by lowercased email, calls `user.comparePassword(password)`. Returns `{ token, profile }` or `401`.

#### `POST /api/auth/guest`

Creates a throwaway user with a unique generated email (`guest_{timestamp}@nutrican.guest`), no password, `isGuest: true`. Returns `{ token, profile }`. Since guest users accumulate in the database, a background cleanup job (not yet implemented) would be needed in production.

#### `GET /api/auth/me`

Protected route. Simply returns `{ profile: req.user }`. Used by the frontend on startup to validate a stored JWT and restore session state.

#### `GET /api/profile`

Returns `{ profile: req.user }` — the current user's profile.

#### `PATCH /api/profile`

Accepts a whitelist of allowed fields:
```
name, age, height, weight, cancerType, cancerStage,
otherConditions, treatmentStages, plan, isVerified,
documentsSubmitted, trialStartedAt, bmi
```
Email and password are not patchable through this endpoint. Uses `findByIdAndUpdate` with `runValidators: true`.

#### `POST /api/profile/upgrade`

Sets `plan: 'Premium'` on the user. No payment processing in current implementation (demo upgrade).

#### `GET /api/journal`

Returns the user's last 30 journal entries sorted by `createdAt` descending. Shapes each entry into the frontend `JournalEntry` interface:
- `id` ← `_id.toString()`
- `timestamp` ← `createdAt.toISOString()`
- `name` ← short weekday string from the creation date

#### `POST /api/journal`

Validates `weight` and `energy` (required). Optionally accepts `bp` (must be a number, not NaN) and `notes` (trimmed). Returns `{ id }`.

#### `GET /api/meals`

Returns the user's last 50 logged meals sorted by date descending.

#### `POST /api/meals`

Validates `name` and `nutrients` are present. Returns `{ id }`.

#### `POST /api/documents/upload`

Multipart form upload. Uses `multer` with in-memory storage (no temp files on disk) and:
- **File size limit**: 10 MB
- **MIME type filter**: Only `application/pdf` allowed

Uploads the buffer to DigitalOcean Spaces using the key:
```
users/{userId}/documents/{timestamp}-{originalFilename}
```
Documents are stored with `ACL: 'private'` — never publicly accessible without a signed URL.

After upload, creates a `Document` record and auto-verifies the user (`isVerified: true`, `documentsSubmitted: true`) for demo purposes.

#### `GET /api/documents`

Returns all of the user's documents. For each document, generates a **15-minute pre-signed S3 URL** (`GetObjectCommand` via `@aws-sdk/s3-request-presigner`) appended as `downloadUrl`. This pattern ensures medical documents are never publicly accessible.

### 4.8 AI Integration (NutriCan ML Model)

All AI endpoints are in `routes/ai.js`. The entire router is behind `router.use(requireAuth)`, so unauthenticated requests are rejected before any ML Model call is made.

**NutriCan ML Model client initialisation:**
```javascript
const nutricanML = new NutriCanML({ apiKey: process.env.NUTRICAN_ML_API_KEY });
const MODEL = 'nutrican-ml-model';
```

**`callModelJSON(prompt, systemPrompt)`** — Core helper:
```javascript
nutricanML.chat.completions.create({
  model: MODEL,
  messages,
  temperature: 0.4,
  response_format: { type: 'json_object' },   // Forces valid JSON output
})
```
Using `response_format: { type: 'json_object' }` prevents the model from wrapping the JSON in markdown fences, ensuring `JSON.parse()` always succeeds on valid responses.

#### AI Endpoint Details

| Route | Model Temperature | Prompt Strategy | Validation |
|---|---|---|---|
| `POST /api/ai/food-safety` | 0.4 | Short single-turn, JSON with `status`/`reason` | Checks `status` is one of three valid values |
| `POST /api/ai/meal-plan` | 0.4 | Long structured prompt, forced Ugandan food list, BMI context | Checks `weekPlan` is an array of exactly 7 items |
| `POST /api/ai/swap-meal` | 0.4 | Replacement-focused prompt, excludes current meal | Checks `name`, `description`, `category` present |
| `POST /api/ai/nutrient-info` | 0.4 | Minimal nutrition analysis prompt | Checks all three numbers are of type `number` |
| `POST /api/ai/symptom-tips` | 0.4 | Symptom-specific food recommendations | Checks `recommendations` is an array |
| `POST /api/ai/doctor-chat` | 0.7 | Full system prompt with doctor persona + patient context, chat history | N/A — raw text reply |

**Doctor chat note:** Unlike other endpoints, `doctor-chat` does NOT use `response_format: json_object`. It builds a proper multi-turn message array with system prompt, conversation history (mapped from frontend `ChatMessage[]`), and the new user message, then returns a plain text reply. Max tokens is set to 300 to keep responses concise.

**Error handling:** Every endpoint catches NutriCan ML Model errors and returns safe fallback responses (e.g., `status: 'Avoid'` for food safety, empty recommendations for symptom tips) rather than propagating raw errors to the client.

### 4.9 File Storage (DigitalOcean Spaces)

DigitalOcean Spaces is an S3-compatible object storage service. The backend uses the official AWS SDK v3 pointing at the Spaces endpoint.

```javascript
const s3 = new S3Client({
  endpoint: process.env.DO_SPACES_ENDPOINT,   // e.g. https://nyc3.digitaloceanspaces.com
  region: process.env.DO_SPACES_REGION,       // e.g. nyc3
  credentials: {
    accessKeyId: process.env.DO_SPACES_KEY,
    secretAccessKey: process.env.DO_SPACES_SECRET,
  },
  forcePathStyle: false,    // DigitalOcean requires virtual-hosted style
});
```

**Upload flow:**
1. `multer.memoryStorage()` buffers the file in RAM (never written to disk).
2. `PutObjectCommand` sends the buffer directly to Spaces with `ACL: 'private'`.
3. A `Document` record is created in MongoDB with the `spacesKey` for later retrieval.

**Download flow:**
1. `GetObjectCommand` is signed with `expiresIn: 900` (15 minutes).
2. The signed URL is returned to the frontend as `downloadUrl`.
3. After 15 minutes the link expires, requiring a fresh API call to get a new one.

## 10. Security Model

### Authentication
- Passwords are hashed with **bcrypt** at cost factor 12 before storage. Plaintext passwords are never stored or logged.
- JWTs are signed with `HS256` using a secret from `JWT_SECRET`. Tokens expire after 7 days.
- The `passwordHash` field is stripped from all API responses via the Mongoose `toJSON` transform.
- All authenticated routes validate the JWT server-side on every request for freshness (no client-side trust).

### Data Isolation
- Every database query for journal entries, meals, and documents filters by `userId: req.user._id`. A user cannot access another user's data even with a valid JWT.

### File Security
- Medical documents are stored with `ACL: 'private'`. They are never publicly accessible via a static URL.
- Access requires a fresh 15-minute pre-signed URL generated server-side on every `GET /api/documents` call.
- Only PDF files are accepted (`fileFilter` in multer). The maximum file size is 10 MB.
- Files are stored in-memory (never written to the server's disk) before being streamed to Spaces.

### CORS
- In production, only explicitly listed origins in `ALLOWED_ORIGINS` can call the API.
- In development, any `localhost` origin is permitted for convenience.

### Input Validation
- All route handlers validate required fields before touching the database.
- The profile `PATCH` endpoint uses an explicit whitelist of allowed fields, preventing mass-assignment of sensitive fields like `email` or `passwordHash`.
- The AI routes validate that Groq responses match the expected schema before returning them, preventing injection of unexpected data into the frontend.

### Legacy Firestore Rules
The `firestore.rules` file in the project root is a remnant of a prior Firebase implementation. It is **not active** in the current architecture, which uses MongoDB + JWT. It can be safely ignored or removed.
